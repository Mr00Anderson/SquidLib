007/**
008 * A quasi-random number generator that goes through one of many sub-random sequences found by J.G. van der Corput.
009 * More specifically, this offers both the normal van der Corput sequence, which only changes the state by incrementing
010 * it (this works better in the normal usage as part of a 2D or 3D point generator) and a kind of scrambled van der
011 * Corput sequence, where the state changes unpredictably (this works better when using this to generate 1D sequences,
012 * or when the base may be larger than 7 or non-prime). In both cases, the state is internally stored in a 64-bit long
013 * that is incremented once per generated number, but when scramble is true, the state is altered with something similar
014 * to a Gray code before being used; more on this later if you want to read about it. The important things to know about
015 * this class are: size of state affects speed (prefer smaller seeds, but quality is sometimes a bit poor at first if
016 * you start at 0); the base (when given) should be prime (smaller bases tend to yield better quality, but oddly, larger
017 * bases perform better); this doesn't generate very random numbers when scramble is false (which can be good
018 * for making points that should not overlap), but it will seem much more random when scramble is true; this is a
019 * StatefulRandomness with an additional method for generating quasi-random doubles, {@link #nextDouble()}; and there
020 * are several static methods offered for convenient generation of points on the related Halton sequence (as well as
021 * faster generation of doubles in the base-2 van der Corput sequence, and a special method that switches which base
022 * it uses depending on the index to seem even less clearly-patterned).
023 * <br>
024 * This generator allows a base (also called a radix) that changes the sequence significantly; a base should be prime,
025 * and this performs better in terms of time used with larger primes, though quality is also improved by preferring
026 * primes that aren't very large relative to the quantity of numbers expected to be generated. Unfortunately,
027 * performance is not especially similar to conventional PRNGs; smaller (positive) state values are processed more
028 * quickly than larger ones, or most negative states. At least one 64-bit integer modulus and one 64-bit integer
029 * division are required for any number to be produced, and the amount of both of these relatively-non-cheap operations
030 * increases linearly with the number of digits (in the specified base, which is usually not 10) needed to represent
031 * the current state. Since performance is not optimal, and the results are rather strange anyway relative to PRNGs,
032 * this should not be used as a direct substitute for a typical RandomnessSource (however, it is similar to, but simpler
033 * and faster than, {@link SobolQRNG}). So what's it good for?
034 * <br>
035 * A VanDerCorputSequence can be a nice building block for more complicated quasi-randomness, especially for points in
036 * 2D or 3D, when scramble is false. There's a simple way that should almost always "just work" as the static method
037 * {@link #halton(int, int, int, int[])} here. If it doesn't meet your needs, there's a little more complexity involved.
038 * Using a VanDerCorputQRNG with base 3 for the x-axis and another VanDerCorputQRNG with base 5 for the y-axis,
039 * requesting a double from each to make points between (0.0, 0.0) and (1.0, 1.0), has an interesting trait that can be
040 * desirable for many kinds of positioning in 2D: once a point has been generated, an identical point will never be
041 * generated until floating-point precision runs out, but more than that, nearby points will almost never be generated
042 * for many generations, and most points will stay at a comfortable distance from each other if the bases are different
043 * and both prime (or, more technically, if they share no common denominators). This is also known as a Halton sequence,
044 * which is a group of sequences of points instead of simply numbers. The choices of 3 and 5 are examples; any two
045 * different primes will technically work for 2D (as well as any two numbers that share no common factor other than 1,
046 * that is, they are relatively coprime), but patterns can be noticeable with primes larger than about 7, with 11 and 13
047 * sometimes acceptable. Three VanDerCorputQRNG sequences could be used for a Halton sequence of 3D
048 * points, using three different prime bases, four for 4D, etc. SobolQRNG can be used for the same purpose, but the
049 * points it generates are typically more closely-aligned to a specific pattern, the pattern is symmetrical between all
050 * four quadrants of the square between (0.0, 0.0) and (1.0, 1.0) in 2D, and it probably extends into higher dimensions.
051 * Using one of the possible Halton sequences gives some more flexibility in the kinds of random-like points produced.
052 * Oddly, using a base-2 van der Corput sequence as the x axis in a Halton sequence and a base-39 van der Corput
053 * sequence as the y axis results in the greatest minimum distance between points found so far while still involving a
054 * base-2 sequence (which is preferential for performance). There, the minimum distance between the first 65536 points
055 * in the [0,1) range is 0.001147395; a runner-up is a y base of 13, which has a minimum distance of 0.000871727 . The
056 * (2,39) Halton sequence is used by {@link #halton(int, int, int)} and {@link #halton(int, int, int, int[])}.
057 * <br>
058 * Because just using the state in a simple incremental fashion puts some difficult requirements on the choice of base
059 * and seed, we can scramble the long state {@code i} with code equivalent to
060 * {@code (i ^ Long.rotateLeft(i, 7) ^ Long.rotateLeft(i, 17))}. This scramble is different from earlier versions and
061 * unlike the earlier ones has a one-to-one mapping from input states to scrambled states.
062 * <br>
063 * Created by Tommy Ettinger on 11/18/2016. Uses code adapted from
064 * <a href="https://blog.demofox.org/2017/05/29/when-random-numbers-are-too-random-low-discrepancy-sequences/">Alan Wolfe's blog</a>,
065 * which turned out to be a lot faster than the previous way I had it implemented.
066 */
067public class VanDerCorputQRNG implements StatefulRandomness, RandomnessSource, Serializable {

095    /**
096     * Constructs a new van der Corput sequence generator with the given base (a.k.a. radix; if given a base less than
097     * 2, this will use base 2 instead) and starting point in the sequence as a seed. Good choices for base are between
098     * 10 and 60 or so, and should usually be prime. Good choices for seed are larger than base but not by very much,
099     * and should generally be positive at construction time.
100     * @param base the base or radix used for this VanDerCorputQRNG; for most uses this should be prime but small-ish
101     * @param seed the seed as a long that will be used as the starting point in the sequence; ideally positive but low
102     * @param scramble if true, will produce more-random values that are better for 1D; if false, better for 2D or 3D
103     */
104    public VanDerCorputQRNG(int base, long seed, boolean scramble)
105    {
106        this.base = base < 2 ? 2 : base;
107        state = seed;
108        this.scramble = scramble;
109    }

111    /**
112     * Gets the next quasi-random long as a fraction of {@link Long#MAX_VALUE}; this can never produce a negative value.
113     * It is extremely unlikely to produce two identical values unless the state is very high or is negative; state
114     * increases by exactly 1 each time this, {@link #next(int)}, or {@link #nextDouble()} is called and can potentially
115     * wrap around to negative values after many generations.
116     * @return a quasi-random non-negative long; may return 0 rarely, probably can't return {@link Long#MAX_VALUE}
117     */
118    @Override
119    public long nextLong() {
120        long n = (((scramble) ? (++state ^ (state << 7 | state >>> 57) ^ (state << 17 | state >>> 47)) : ++state)
121                & 0x7fffffffffffffffL);
122        if(base <= 2) {
123            return Long.reverse(n) >>> 1;
124        }
125        double denominator = base, res = 0.0;
126        while (n > 0)
127        {
128            res += (n % base) / denominator;
129            n /= base;
130            denominator *= base;
131        }
132        return (long) (res * Long.MAX_VALUE);
133    }
134
135    @Override
136    public int next(int bits) {
137        return (int)(nextLong()) >>> (32 - bits);
138    }
139
140    /**
141     * Gets the next quasi-random double from between 0.0 and 1.0 (normally both exclusive; only if state is negative or
142     * has wrapped around to a negative value can 0.0 ever be produced). It should be nearly impossible for this to
143     * return the same number twice unless floating-point precision has been exhausted or a very large amount of numbers
144     * have already been generated. Certain unusual bases may make this more likely, and similar numbers may be returned
145     * more frequently if scramble is true.
146     * @return a quasi-random double that will always be less than 1.0 and will be no lower than 0.0
147     */
148    public double nextDouble() {
149        long n = (((scramble) ? (++state ^ (state << 7 | state >>> 57) ^ (state << 17 | state >>> 47)) : ++state)
150                & 0x7fffffffffffffffL);
151        if(base <= 2) {
152            return (Long.reverse(n) >>> 11) * 0x1p-53;
153        }
154        double denominator = base, res = 0.0;
155        while (n > 0)
156        {
157            res += (n % base) / denominator;
158            n /= base;
159            denominator *= base;
160        }
161        return res;
162    }

197    @Override
198    public int hashCode() {
199        int result = (int) (state ^ (state >>> 32));
200        result = 31 * result + base | 0;
201        return 31 * result + (scramble ? 1 : 0) | 0;
202    }

204    /**
205     * Convenience method that gets a quasi-random 2D point between integer (0,0) inclusive and (width,height)
206     * exclusive and fills it into point. This is roughly equivalent to creating two VanDerCorputQRNG generators, one
207     * with {@code new VanDerCorputQRNG(2, index, false)} and the other with
208     * {@code new VanDerCorputQRNG(39, index, false)}, then getting an x-coordinate from the first with
209     * {@code (int)(nextDouble() * width)} and similarly for y with the other generator. The advantage here is you don't
210     * actually create any objects using this static method, only assigning to point, if valid. You might find an
211     * advantage in using values for index that start higher than 20 or so, but you can pass sequential values for index
212     * and generally get points that won't be near each other; this is not true for all parameters to Halton sequences,
213     * but it is true for this one.
214     * @param width the maximum exclusive bound for the x-positions (index 0) of points this can return
215     * @param height the maximum exclusive bound for the y-positions (index 1) of points this can return
216     * @param index an int that, if unique, positive, and not too large, will usually result in unique points
217     * @param point an int array that will be modified; should have length 2; if null or too small, a new array will be created
218     * @return point after modifications to the first two items, or a new array if point is null or too small
219     */
220    public static int[] halton(int width, int height, int index, int[] point)
221    {
222        if(point == null || point.length < 2)
223            point = new int[2];
224
225        double denominator = 39.0, resY = 0.0;
226        int n = (index+1 & 0x7fffffff);
227        while (n > 0)
228        {
229            resY += (n % 39) / denominator;
230            n /= 39;
231            denominator *= 39.0;
232        }
233        point[0] = (int)(width * (Integer.reverse(index + 1) >>> 1) * 0x1p-31);
234        point[1] = (int)(resY * height);
235        return point;
236    }

255    /**
256     * Convenience method that gets a quasi-random Coord between integer (0,0) inclusive and (width,height) exclusive
257     * and gets the corresponding Coord from the Coord pool. This is roughly equivalent to creating two VanDerCorputQRNG
258     * generators, one with {@code new VanDerCorputQRNG(2, index, false)} and the other with
259     * {@code new VanDerCorputQRNG(39, index, false)}, then getting an x-coordinate from the first with
260     * {@code (int)(nextDouble() * width)} and similarly for y with the other generator. You might find an advantage in
261     * using values for index that start higher than 20 or so, but you can pass sequential values for index and
262     * generally get points that won't be near each other; this is not true for all parameters to Halton sequences, but
263     * it is true for this one.
264     * @param width the width of the area this can cover
265     * @param height the height of the area this can cover
266     * @param xOffset the lowest x-coordinate this can produce, and also added to width to get the upper bound on x
267     * @param yOffset the lowest y-coordinate this can produce, and also added to height to get the upper bound on y
268     * @param index an int that, if unique, positive, and not too large, will usually result in unique points
269     * @return point after modifications to the first two items, or a new array if point is null or too small
270     */
271    public static Coord halton(int width, int height, int xOffset, int yOffset, int index)
272    {
273        double denominator = 39.0, resY = 0.0;
274        int n = (index+1 & 0x7fffffff);
275        while (n > 0)
276        {
277            resY += (n % 39) / denominator;
278            n /= 39;
279            denominator *= 39.0;
280        }
281        return Coord.get((int)(width * (Integer.reverse(index + 1) >>> 1) * 0x1p-31) + xOffset, (int)(resY * height) + yOffset);
282    }

284    /**
285     * Convenience method that gets a quasi-random 3D point between integer (0,0,0) inclusive and (width,height,depth)
286     * exclusive. This is roughly equivalent to creating three VanDerCorputQRNG generators, one with
287     * {@code new VanDerCorputQRNG(2, index, false)} another with {@code new VanDerCorputQRNG(3, index, false)},
288     * and another with {@code new VanDerCorputQRNG(5, index, false)}, then getting an x-coordinate from the first with
289     * {@code (int)(nextDouble() * width)} and similarly for y and z with the other generators. The advantage here is
290     * you don't actually create any objects using this static method, only assigning to point, if valid. You might find
291     * an advantage in using values for index that start higher than 20 or so, but you can pass sequential values for
292     * index and generally get points that won't be near each other; this is not true for all parameters to Halton
293     * sequences, but it is true for this one.
294     * @param width the maximum exclusive bound for the x-positions (index 0) of points this can return
295     * @param height the maximum exclusive bound for the y-positions (index 1) of points this can return
296     * @param depth the maximum exclusive bound for the z-positions (index 2) of points this can return
297     * @param index an int that, if unique, positive, and not too large, will usually result in unique points
298     * @param point an int array that will be modified; should have length 3; if null or too small, a new array will be created
299     * @return point after modifications to the first two items, or a new array if point is null or too small
300     */
301    public static int[] halton(int width, int height, int depth, int index, int[] point)
302    {
303        if(point == null || point.length < 3)
304            point = new int[3];
305
306        double denominator = 3.0, resY = 0.0, resZ = 0.0;
307        int n = (index+1 & 0x7fffffff);
308        while (n > 0)
309        {
310            resY += (n % 3) / denominator;
311            n /= 3;
312            denominator *= 3.0;
313        }
314
315        denominator = 5;
316        n = (index+1 & 0x7fffffff);
317        while (n > 0)
318        {
319            resZ += (n % 5) / denominator;
320            n /= 5;
321            denominator *= 5.0;
322        }
323        point[0] = (int)(width * (Integer.reverse(index + 1) >>> 1) * 0x1p-31);
324        point[1] = (int)(resY * height);
325        point[2] = (int)(resZ * depth);
326        return point;
327    }

330     * Convenience method that gets a quasi-random Coord3D between integer (0,0,0) inclusive and (width,height,depth)
331     * exclusive. This is roughly equivalent to creating three VanDerCorputQRNG generators, one with
332     * {@code new VanDerCorputQRNG(2, index, false)} another with {@code new VanDerCorputQRNG(3, index, false)},
333     * and another with {@code new VanDerCorputQRNG(5, index, false)}, then getting an x-coordinate from the first with
334     * {@code (int)(nextDouble() * width)} and similarly for y and z with the other generators. This overload always
335     * creates a new Coord3D object, so you might prefer {@link #halton(int, int, int, int, int[])}, which can reuse an
336     * int array. You might find an advantage in using values for index that start higher than 20 or so, but you can
337     * pass sequential values for index and generally get points that won't be near each other; this is not true for all
338     * parameters to Halton sequences, but it is true for this one.
339     * @param width the maximum exclusive bound for the x-positions (index 0) of points this can return
340     * @param height the maximum exclusive bound for the y-positions (index 1) of points this can return
341     * @param depth the maximum exclusive bound for the z-positions (index 2) of points this can return
342     * @param index an int that, if unique, positive, and not too large, will usually result in unique points
343     * @return a new Coord3D with x,y,z between 0,0,0 (inclusive) and width,height,depth (exclusive)
344     */
345    public static Coord3D halton(int width, int height, int depth, int index)
346    {
347        double denominator = 3.0, resY = 0.0, resZ = 0.0;
348        int n = (index+1 & 0x7fffffff);
349        while (n > 0)
350        {
351            resY += (n % 3) / denominator;
352            n /= 3;
353            denominator *= 3.0;
354        }
355
356        denominator = 5;
357        n = (index+1 & 0x7fffffff);
358        while (n > 0)
359        {
360            resZ += (n % 5) / denominator;
361            n /= 5;
362            denominator *= 5.0;
363        }
364        return new Coord3D((int)(width * (Integer.reverse(index + 1) >>> 1) * 0x1p-31),
365                (int)(resY * height),
366                (int)(resZ * depth));
367    }

369    /**
370     * Convenience method to get a double from the van der Corput sequence with the given {@code base} at the requested
371     * {@code index} without needing to construct a VanDerCorputQRNG. You should use a prime number for base; 2, 3, 5,
372     * and 7 should be among the first choices to ensure optimal quality unless you are scrambling the index yourself.
373     * If speed is the priority, then larger prime bases counter-intuitively perform better than small ones; 0x1337,
374     * 0xDE4D, 0x510B and 0xACED are all probable primes (using {@link java.math.BigInteger#isProbablePrime(int)}) that
375     * may do well here for speed but will likely require some basic scrambling of the index order. This method on its
376     * own does not perform any scrambling on index other than incrementing it and ensuring it is positive (by
377     * discarding the sign bit; for all positive index values other than 0x7FFFFFFF, this has no effect). If you want
378     * to quickly scramble an int index {@code i} for this purpose, try
379     * {@code (i ^ (i << 7 | i >>> 25) ^ (i << 19 | i >>> 13))}, which may compile to SSE instructions on recent 
380     * desktop processors and won't risk losing precision on GWT.
381     * <br>
382     * Uses the same algorithm as {@link #determine2(int)} when base is 2, which should offer some speed improvement.
383     * The other bases use code adapted from
384     * <a href="https://blog.demofox.org/2017/05/29/when-random-numbers-are-too-random-low-discrepancy-sequences/">Alan Wolfe's blog</a>,
385     * which turned out to be a lot faster than the previous way I had it implemented.
386     * @param base a prime number to use as the base/radix of the van der Corput sequence
387     * @param index the position in the sequence of the requested base, as a non-negative int
388     * @return a quasi-random double between 0.0 (inclusive) and 1.0 (exclusive).
389     */
390    public static double determine(final int base, final int index)
391    {
392        if(base <= 2) {
393            return (Integer.reverse(index + 1) >>> 1) * 0x1p-31;
394        }
395        double denominator = base, res = 0.0;
396        int n = (index+1 & 0x7fffffff);
397        while (n > 0)
398        {
399            res += (n % base) / denominator;
400            n /= base;
401            denominator *= base;
402        }
403        return res;
404    }

406    /**
407     * Convenience method to get a double from the van der Corput sequence with the base 2 at the requested
408     * {@code index} without needing to construct a VanDerCorputQRNG. This does not perform any scrambling on index
409     * other than incrementing it and ensuring it is positive (by discarding the sign bit; for all positive index values
410     * other than 0x7FFFFFFF ({@link Integer#MAX_VALUE}), this has no effect).
411     * <br>
412     * Because binary manipulation of numbers is easier and more efficient, the technique used by this method is also
413     * used by {@link #determine(int, int)} when base is 2, and should be faster than other bases.
414     * @param index the position in the base-2 van der Corput sequence, as a non-negative int
415     * @return a quasi-random double between 0.0 (inclusive) and 1.0 (exclusive).
416     */
417    public static double determine2(int index)
418    {
419        return (Integer.reverse(index + 1) >>> 1) * 0x1p-31;
420    }
421    /**
422     * Method to get a double from the van der Corput sequence with the base 2 at a scrambling of the requested
423     * {@code index} without needing to construct a VanDerCorputQRNG. This performs different scrambling on index
424     * than the instance methods on this class perform, and it seems to do well enough while being a little simpler.
425     * This is meant to be usable as an alternative to a different base for a van der Corput sequence when you need two
426     * different sequences, and are already using base 2 via {@link #determine2(int)}.
427     * <br>
428     * Because binary manipulation of numbers is easier and more efficient, this method should be somewhat faster than
429     * the alternatives, like {@link #determine(int, int)} with base 2. It should take only slightly longer to run than
430     * {@link #determine2(int)}, due to the brief amount of time needed to scramble the index.
431     * @param index the position in the sequence of the requested base
432     * @return a quasi-random double between 0.0 (inclusive) and 1.0 (exclusive).
433     */
434    public static double determine2_scrambled(int index)
435    {
436        return (Integer.reverse(++index ^ (index << 7 | index >>> 25) ^ (index << 19 | index >>> 13)) >>> 1) * 0x1p-31;
437    }

441    /**
442     * Chooses one sequence from the van der Corput sequences with bases 2, 3, and 5, where 5 is used 1/8 of the time,
443     * 3 is used 3/8 of the time, and 2 is used 1/2 of the time, and returns a double from the chosen sequence at the
444     * specified {@code index}. The exact setup used for the choice this makes is potentially fragile, but in certain
445     * circumstances this does better than {@link SobolQRNG} at avoiding extremely close values (the kind that overlap
446     * on actual maps). Speed is not a concern here; this should be very much fast enough for the expected usage in
447     * map generation (it's used in {@link GreasedRegion#quasiRandomSeparated(double)}.
448     * @param index the index to use from one of the sequences; will also be used to select sequence
449     * @return a double from 0.0 (inclusive, but extremely rare) to 1.0 (exclusive); values will tend to spread apart
450     */
451    public static double determineMixed(int index)
452    {
453        return determine(lowPrimes[index & 7], index);
454    }

456    /**
457     * Given any int (0 is allowed), this gets a somewhat-sub-random float from 0.0 (inclusive) to 1.0 (exclusive)
458     * using the same implementation as {@link NumberTools#randomFloat(long)} but with index alterations. Only "weak"
459     * because it lacks the stronger certainty of subsequent numbers being separated that the Van der Corput sequence

461     * algorithm, which does not guarantee that its outputs are unique).

463     * Not all int values for index will produce unique results, since this produces a float and there are less distinct
464     * floats between 0.0 and 1.0 than there are all ints (1/512 as many floats in that range as ints, specifically).
465     * It should take a while calling this method before you hit an actual collision.
466     * @param index any int
467     * @return a float from 0.0 (inclusive) to 1.0 (exclusive) that should not be closely correlated to index
468     */

470    {
471        return NumberTools.randomFloat((index >>> 19 | index << 13) ^ 0x13A5BA1D);

492     * Halton-like sequence with almost-as-good distance between points. The base is allowed to be any odd long,

494     * return any specific number (it will vary with the base). This returns a double between 0.0 inclusive and 1.0

501     * sometimes be useful; when you want optimal distance and don't have a case where a clear pattern on a grid is an

512    /**
513     * A quasi-random number generator of doubles between 0.0 inclusive and 1.0 exclusive, but that has issues when it
514     * would be used like a Halton sequence. Only ideal in 1D, this produces well-separated points that are aligned to
515     * parallel hyperplanes when called with different bases and each base used as an axis for more than 1 dimension.
516     * This can produce points with more separation in 2D than a Halton sequence can, but not often. Two bases that do
517     * this when used together are the ints 0xDE4DBEEF and 0x1337D00D (or in decimal, -565330193 and 322424845; note
518     * that 0xDE4DBEEF is a negative integer); they were tried as a gimmick but nothing else turned out better. They do
519     * still produce points on parallel lines, and like all bases, never points between those lines.
520     * <br>
521     * Note, the source of this method is one line, and you may see benefits from copying that code into the call-site
522     * with minor modifications. This returns
523     * {@code (((base * Integer.reverse(index)) << 21) & 0x1fffffffffffffL) * 0x1p-53;}, where base is a long and index
524     * is an int (as in the method signature). The multiplier 0x1p-53 is a very small hexadecimal double literal, using
525     * the same syntax as other parts of SquidLib use for packed floats; using this helps avoid precision loss. If you
526     * want a range larger than 0.0 to 1.0, you can change the multiplier {@code 0x1p-53} to some other constant, like
527     * declaring {@code final double upTo100 = 0x1p-53 * 100.0} before some code that wants quasi-random numbers between
528     * 0.0 inclusive and 100.0 exclusive, then using
529     * {@code (((base * Integer.reverse(index)) << 21) & 0x1fffffffffffffL) * upTo100;} to get those numbers. It isn't
530     * precise to use this technique to get numbers with an upper bound less than 1.0, because 0x1p-53 is about as small
531     * as a double can get with precision intact. In that case, you can use
532     * {@code (((base * Integer.reverse(index)) << 8) & 0xffffffffffL) * smallBound;} where smallBound has been declared
533     * as {@code final double smallBound = 0x1p-40 * 0.05;} (where 0.05 can be switched out for any double between
534     * 1.0/8192.0 and 1.0).
535     * @param base any odd long; the most significant 21 bits (except the sign bit) are effectively ignored
536     * @param index any int; if 0 this will return 0
537     * @return a double between 0.0 inclusive and 1.0 exclusive
538     */
539    public static double planarDetermine(long base, final int index) { return ((base * Integer.reverse(index) << 21) & 0x1fffffffffffffL) * 0x1p-53; }

541    /**
542     * Samples a quasi-random Coord sequence that is almost unique to the given seed, getting a Coord with x between
543     * xOffset (inclusive) and width + xOffset (exclusive), y between yOffset (inclusive) and height + yOffset
544     * (exclusive). The seed is "almost" unique because even seeds are discouraged; there is an identical sequence for
545     * every even seed produced by some odd seed. This generates a not-very random number, reverses its bits as other
546     * methods in this class do, then treats that single 32-bit int as two coordinates on a Z-order curve to get
547     * separate x and y from them. In practice these Coords are very well-dispersed if only a small amount are sampled
548     * and all the index values are close-by, and get closer together as more are sampled. Unlike the Halton sequence,
549     * this has very different results with different seeds (Halton only allows bases to be changed), and doesn't
550     * involve any division, modulus, conditionals, or loops.
551     * @param seed an int seed that should be an odd number
552     * @param width the width of the area this can cover
553     * @param height the height of the area this can cover
554     * @param xOffset the lowest x-coordinate this can produce, and also added to width to get the upper bound on x
555     * @param yOffset the lowest y-coordinate this can produce, and also added to height to get the upper bound on y
556     * @param index the index in the sequence, almost always a positive int that increases by 1 with each call
557     * @return a Coord between (xOffset, yOffset) inclusive and (width+xOffset, height+yOffset) exclusive
558     */
559    public static Coord haltoid(int seed, int width, int height, int xOffset, int yOffset, int index)
560    {
561        int morton = GreasedRegion.disperseBits(Integer.reverse((seed * 0x2C9277B5 | 1) * (index + 1)));
562        return Coord.get((int)(width * ((morton & 0xffff) * 0x1p-16)) + xOffset, (int)(((morton >>> 16) * 0x1p-16) * height) + yOffset);
563    }
564    /**
565     * Martin Roberts' "unreasonably effective" quasi-random int sequence based on the golden ratio.
566     * See <a href="http://extremelearning.com.au/unreasonable-effectiveness-of-quasirandom-sequences/">his blog</a> for
567     * more detailed info, but this can be summarized as being extremely good at separating outputs at the expense of
568     * seeming less random. Produces an int between offset (inclusive) and offset + span (exclusive), with the int at
569     * each {@code index} likely to be different for at least {@code span / 4} indices (very low spans may offer less of
570     * a guarantee).
571     * <br>
572     * Note, use {@link #roberts(int, int, int, int, int)} for 2D points and this method for 1D values; the same
573     * properties won't be preserved if you use a 1D Roberts sequence in 2D.
574     * @param span the size of the range this can return
575     * @param offset the minimum value this can return
576     * @param index the index of the int in the 1D Roberts sequence; should be greater than 0, but not required to be
577     * @return an int between offset inclusive and offset+span exclusive
578     */
579    public static int roberts(int span, int offset, int index)
580    {
581        return (int)((index * 0x9E3779B9L & 0xFFFFFFFFL) * span >>> 32) + offset;
582    }
