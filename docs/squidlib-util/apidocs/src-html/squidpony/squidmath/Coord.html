007/**
008 * A 2D coordinate with (constant) x and y fields. Coord objects are immutable; a single pool of Coord values, with
009 * x and y each ranging from -3 to 255, is shared by all users of Coord. This pool helps reduce pressure on the
010 * garbage collector when many Coord values would have been created for some purpose and quickly discarded; instead
011 * of creating a new Coord with a constructor, you use the static method {@link #get(int, int)}, which retrieves an
012 * already-existing Coord from the pool if possible, and always returns a usable Coord.
013 * <br>
014 * The Coord class is a fundamental part of SquidLib; any class that uses positions on a grid makes use of it here.
015 * It finds usage naturally in classes throughout {@link squidpony.squidgrid}, with {@link squidpony.squidgrid.zone}
016 * providing an abstraction around groups of Coord and {@link squidpony.squidgrid.iterator} providing various ways to
017 * iterate through the Coords that make up a larger shape. In this package, {@link squidpony.squidmath}, a few classes
018 * should be pointed out. {@link CoordPacker} is a class with all static methods that provides various ways to compress
019 * the memory usage of regions made of many Coord values (and can be constructed in other ways but still provide Coords
020 * later), but since Coords don't use much memory anyway, the real use of the class is for manipulating the shapes and
021 * sizes of the regions those Coords are part of. {@link GreasedRegion} has similar functionality to CoordPacker, but
022 * where CoordPacker is purely static, taking and returning regions as encoded, usually-low-memory-cost arrays of
023 * {@code short} that it considers immutable, a GreasedRegion is a mutable object that allows the same region-altering
024 * techniques to be applied in-place in a way that is relatively (very) low-time-cost. If deciding between the two,
025 * GreasedRegion should usually be preferred, and CoordPacker cannot actually be used when storing regions in larger
026 * than a 256x256 space (usually when the Coord pool has been expanded; see below); GreasedRegion can store potentially
027 * large positions.
028 * <br>
029 * More on the Coord pool used by this class:  Coords can't always be retrieved from the pool; Coord.get constructs a
030 * new Coord if one of x or y is unusually large (greater than 255) or too negative (below -3). The upper limit of 255
031 * is not a hard rule; you can increase the limit on the pool by calling {@link #expandPoolTo(int, int)} or
032 * {@link #expandPool(int, int)}, which cause more memory to be spent initially on storing Coords but can save memory
033 * or ease GC pressure over the long term by preventing duplicate Coords from being created many times. The pool can
034 * never shrink because allowing that would cause completely unpredictable results if existing Coords were in use, or
035 * could easily cause crashes on Android after resuming an application that had previously shrunken the pool due to
036 * platform quirks. Long story short, you should only expand the pool size when your game needs a larger set of 2D
037 * points it will commonly use, and in most cases you shouldn't need to change it at all.
038 * 
039 * Created by Tommy Ettinger on 8/12/2015.
040 */

059    public static Coord get(final int x, final int y)
060    {
061        if(x >= -3 && y >= -3 && x < POOL.length - 3 && y < POOL[x + 3].length - 3)
062            return POOL[x + 3][y + 3];
063        else return new Coord(x, y);
064    }

066        /**
067     * Gets the angle in degrees to go between two Coords; 0 is up.
068         * @param from the starting Coord to measure from
069         * @param to the ending Coord to measure to
070         * @return The degree from {@code from} to {@code to}; 0 is up
071         */
072        public static double degrees(final Coord from, final Coord to) {
073                final int x = to.x - from.x;
074                final int y = to.y - from.y;
075                double degree = Math.toDegrees(NumberTools.atan2(y, x));
076                degree += 450;// rotate to all positive and 0 is up
077                degree %= 360;// normalize
078                return degree;
079        }

335        public boolean isAdjacent(final Coord c) {
336                switch (Math.abs(x - c.x)) {
337                case 0:
338                        return Math.abs(y - c.y) == 1;
339                case 1:
340                        return y == c.y || Math.abs(y - c.y) == 1;
341                default:
342                        return false;
343                }

346        /**
347         * Precondition: {@code this} is {@link #isAdjacent(Coord) adjacent} to
348         * {@code adjacent}.
349     *
350         * @param adjacent
351         *            A {@link Coord} that is {@link #isAdjacent(Coord) adjacent} to
352         *            {@code this}.
353         * @return The direction to go from {@code this} to {@code adjacent} i.e.
354         *         the direction {@code d} such that {@code translate(this, d)}
355         *         yields {@code adjacent}.
356         * @throws IllegalStateException
357         *             If {@code this} isn't adjacent to {@code adjacent}.
358         */
359        /* different implementation than before, closer to isAdjacent() */
360        public Direction toGoTo(final Coord adjacent) {
361        switch (adjacent.x - x) {
362            case 0:
363                switch (adjacent.y - y)
364                {
365                    case -1: return Direction.UP;
366                    case 1: return Direction.DOWN;
367                    default:
368                        throw new IllegalStateException(this + " is not adjacent to " + adjacent);
369                }
370            case 1:
371                switch (adjacent.y - y)
372                {
373                    case -1: return Direction.UP_RIGHT;
374                    case 0: return Direction.RIGHT;
375                    case 1: return Direction.DOWN_RIGHT;
376                    default:
377                        throw new IllegalStateException(this + " is not adjacent to " + adjacent);
378                }
379            case -1:
380                switch (adjacent.y - y)
381                {
382                    case -1: return Direction.UP_LEFT;
383                    case 0: return Direction.LEFT;
384                    case 1: return Direction.DOWN_LEFT;
385                    default:
386                        throw new IllegalStateException(this + " is not adjacent to " + adjacent);
387                }
388            default:
389                throw new IllegalStateException(this + " is not adjacent to " + adjacent);
390        }
391        }

439    /**
440     * Gets the hash code for this Coord; does not use the standard "auto-complete" style of hash that most IDEs will
441     * generate, but instead uses a highly-specific technique based on {@link Lathe32RNG}'s random int generation, which
442     * also has two independent ints it uses like a Coord's x and y. Unlike Lathe32RNG, this uses only bitwise
443     * operations (not even addition). This guarantees it will behave correctly on GWT (Lathe32RNG uses an alternate
444     * source file on GWT).  It also manages to get extremely low collision rates under many circumstances, and very
445     * frequently manages to avoid colliding on more than 25% of Coords (making the load factor of most hash-based
446     * collections fine at a default of 0.75) while often having 0 collisions with some data sets.
447     * <br>
448     * For reference, this was tested on several rectangular sections of Coords of varying density, with sizes from 64
449     * to 512 for x and y separately. On a total of 7,830,980 Coords, an older hashCode() this used got 2,159,553
450     * collisions (27.6% rate), the previous similar hashCode() this used got 1,514,191 collisions (19.3% rate), a naive
451     * {@link java.util.Objects#hash(Object...)} approach on x and y got 6,163,604 collisions (78.7% rate), and this
452     * method gets 179,922 collisions (2.3% rate). A perfect general hashCode isn't possible, but for the common usage
453     * of Coords (usually x and y are no greater than 511 and are rarely negative, etc.) we can do better than
454     * {@code return x * 31 + y;} (which is close to the high-collision Objects.hash way).
455     * <br>
456     * This changed at least five times in SquidLib's history. In general, you shouldn't rely on hashCodes to stay the
457     * same across platforms and versions, whether for the JDK or this library. SquidLib (tries to) never depend on the
458     * unpredictable ordering of some hash-based collections like HashSet and HashMap, instead using its own
459     * {@link OrderedSet} and {@link OrderedMap}; if you use the ordered kinds, then the only things that matter about
460     * this hash code are that it's fast (it's fast enough), it's cross-platform compatible (this version avoids using
461     * long values, which are slow on GWT, and is carefully written to behave the same on GWT as desktop) and that it
462     * doesn't collide often (which is now much more accurate than in earlier versions of this method).
463     * @return an int that should, for most different Coord values, be significantly different from the other hash codes
464     */
465    @Override
466    public int hashCode() {
467        int r = x ^ y;
468        r ^= (x << 13 | x >>> 19) ^ (r << 5) ^ (r << 28 | r >>> 4);
469        r = x ^ (r << 11 | r >>> 21);
470        return r ^ (r << 25 | r >>> 7);
471    }

473    /**
474     * Something like hashCode(), but reversible with {@code Coord.decode()}. Works for Coords between roughly -256 and
475     * 32000 in each of x and y, but will probably only decode to pooled Coords if x and y are both between -3 and 255
476     * (inclusive for both).
477     * @return an int as a unique code for this Coord
478     */
479    public int encode()
480    {
481        return ((x + 256) << 16) ^ (y + 256);
482    }

485     * An alternative to getting a Coord with Coord.get() only to encode() it as the next step. This doesn't create a

487     * @param x the x position to encode
488     * @param y the y position to encode

491    public static int pureEncode(final int x, final int y)

495    /**
496     * This can take an int produced by {@code someCoord.encode()} and get the original Coord back out of it. It
497     * works for all pooled Coords where the pool hasn't been expanded past about 32,000 in either dimension. It even
498     * works for Coords with negative x or y as well, if they are no lower than -256 in either dimension. This will
499     * almost certainly fail (producing a gibberish Coord that probably won't be pooled) on hashes produced by any other
500     * class, including subclasses of Coord.
501     * @param code an encoded int from a Coord, but not a subclass of Coord
502     * @return the Coord that gave hash as its hashCode()
503     */
504    public static Coord decode(final int code)
505    {
506        return get((code >>> 16) - 256, (code & 0xFFFF) - 256);
507    }

557     * Enlarges the pool of cached Coords to the given width and height, and doesn't change
558     * a dimension if it would be reduced in size.
559     * Cached Coord values will be reused by Coord.get instead of re-allocated each time.
560     * The default pool allows Coords with x and y each between -3 and 255, inclusive, to
561     * be cached, and is considered to have width and height of 256 to begin with. Giving a
562     * width greater than 256 will allow Coords with x greater than 255 to be cached;
563     * likewise for height. If width or height is smaller than the current cache width or
564     * height, that dimension will not change, but the other still may if it is valid. You
565     * cannot shrink the pool size.
566     * @param width the new width for the pool of cached Coords; will be ignored if smaller than the current width
567     * @param height the new height for the pool of cached Coords; will be ignored if smaller than the current height
568     */
569    public static void expandPoolTo(final int width, final int height)
570    {
571        expandPool(Math.max(0, width + 3 - POOL.length), Math.max(0, height + 3 - POOL[0].length));
572    }
573
574    /**
575     * Enlarges the pool of cached Coords by the given amount of expansion for x and y.
576     * Cached Coord values will be reused by Coord.get instead of re-allocated each time.
577     * The default pool allows Coords with x and y each between -3 and 255, inclusive, to
578     * be cached, and this can increase the size in the positive direction. If either
579     * xIncrease or yIncrease is negative, this method returns immediately and does nothing
580     * else; the same is true of both arguments are zero. You cannot shrink the pool size.
581     * @param xIncrease the amount to increase cache's width by
582     * @param yIncrease the amount to increase cache's height by
583     */
584    public static void expandPool(final int xIncrease, final int yIncrease)
585    {
586        if(xIncrease < 0 || yIncrease < 0 || (xIncrease | yIncrease) == 0 )
587            return;
588        int width = POOL.length, height = POOL[0].length;
589        Coord[][] POOL2 = new Coord[width + xIncrease][height + yIncrease];
590        for (int i = 0; i < width; i++) {
591            POOL2[i] = new Coord[height + yIncrease];
592            System.arraycopy(POOL[i], 0, POOL2[i], 0, height);
593            for (int j = 0; j < height + yIncrease; j++) {
594                if(POOL2[i][j] == null) POOL2[i][j] = new Coord(i - 3, j - 3);
595            }
596        }
597        for (int i = width; i < width + xIncrease; i++) {
598            POOL2[i] = new Coord[height + yIncrease];
599            for (int j = 0; j < height + yIncrease; j++) {
600                POOL2[i][j] = new Coord(i - 3, j - 3);
601            }
602        }
603        POOL = POOL2;
604    }
605
606    public Coord interpolate(Coord end, float amountTraveled) {
607        return Coord.get(x + Math.round((end.x - x) * amountTraveled),
608                y + Math.round((end.y - y) * amountTraveled));
609    }
