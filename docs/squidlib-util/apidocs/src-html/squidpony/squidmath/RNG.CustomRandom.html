010/**
011 * A wrapper class for working with random number generators in a more friendly way.
012 * Implements {@link IRNG}, which covers most of the API surface, but RNG implements
013 * a decent amount of additional methods. You should consider if your code needs
014 * these additional methods, and if not you should use IRNG as the type for when you
015 * need some random number generator.
016 * <p>
017 * Includes methods for getting values between two numbers and for getting
018 * random elements from a collection or array. There are methods to shuffle
019 * a collection and to get a random ordering that can be applied as one shuffle
020 * across multiple collections, such as via {@link OrderedMap#reorder(int...)},
021 * {@link ArrayTools#reorder(ArrayList, int...)}, and so on. You can construct
022 * an RNG with all sorts of RandomnessSource implementations, and choosing them
023 * is usually not a big concern because the default works very well. If you target
024 * GWT, then it is suggested that you use {@link GWTRNG} instead of RNG; both
025 * implement {@link IRNG}, which is enough for most usage across SquidLib, but
026 * GWTRNG is optimized heavily for better performance on GWT, even returning long
027 * values faster than implementations that natively do their math on longs. It has
028 * worse performance on 64-bit PCs and mobile devices, but should also have better
029 * performance on 32-bit PCs and mobile devices.
030 * <br>
031 * But if you do want advice on what RandomnessSource to use... {@link DiverRNG}
032 * is the default, and is the fastest generator that passes most tests and can
033 * produce all 64-bit values, and though relative to many of the others it has a
034 * significantly shorter period (the amount of random numbers it will go through
035 * before repeating the sequence), this almost never matters in games, and is
036 * primarily relevant for massively-parallel scientific programs. DiverRNG has a
037 * period of {@code pow(2, 64)} as opposed to {@link XoRoRNG}'s
038 * {@code pow(2, 128) - 1}, or {@link LongPeriodRNG}'s {@code pow(2, 1024) - 1}.
039 * {@link LightRNG} is a solid choice and a former default RandomnessSource;
040 * additional features of LightRNG are exposed in {@link MoonwalkRNG} and using
041 * MoonwalkRNG is recommended if you need unusual features like skipping backwards
042 * in a random number sequence, taking a result of a nextLong() call and reversing
043 * it to get the state that produced it, or calculating the distance in number of
044 * nextLong() calls between two results of nextLong() calls. LightRNG is a
045 * StatefulRandomness, which lets it be used in {@link StatefulRNG}, and so is
046 * DiverRNG, but LightRNG is also a {@link SkippingRandomness}, which means you can
047 * leap forward or backwards in its sequence very efficiently (DiverRNG is not a
048 * SkippingRandomness). {@link ThrustAltRNG} provides similar qualities to LightRNG,
049 * and is one of the fastest generators here, but can't produce all possible 64-bit
050 * values (possibly some 32-bit values as well); it was the default at one point so
051 * you may want to keep compatibility with some versions by specifying ThrustAltRNG.
052 * The defaults have changed in the past as issues are found in various generators;
053 * LightRNG has high quality all-around but is slower than the other defaults,
054 * ThrustAltRNG can't produce all results, LinnormRNG passed tests in an earlier
055 * version of the PractRand test suite but now fails in the current version, and now
056 * the default is DiverRNG, which shares the known issue of LightRNG and LinnormRNG
057 * that it can't produce the same result twice from {@link #nextLong()} until the
058 * generator exhausts its period and repeats its output from the beginning.
059 * For most cases, you should decide between DiverRNG, ThrustAltRNG, LightRNG,
060 * LongPeriodRNG, MiniMover64RNG, and XoshiroStarPhi32RNG based on your priorities.
061 * Some tasks are better solved by using a different class, usually {@link GWTRNG},
062 * which can always be serialized on GWT to save its state easily and is usually the
063 * fastest substitute for RNG on that platform. DiverRNG is the best if you want high
064 * speed, very good quality of randomness, and expect to generate a reasonable quantity
065 * of numbers for a game (less than 18446744073709551616 numbers) without any single
066 * results being impossible. LightRNG is the second-best at the above criteria, but is
067 * the best option if you need an RNG that can skip backwards or jump forwards without
068 * incurring speed penalties. LongPeriodRNG is best if you for some reason need a massive
069 * amount of random numbers (as in, ten quintillion would be far too little) or want to
070 * split up such a large generator into unrelated subsequences. XoshiroStarPhi32RNG is
071 * best if GWT is a possible target but you either need to generate more than
072 * 18446744073709551616 numbers (but less than 340282366920938463463374607431768211455
073 * numbers) or you need to ensure that each 128-bit chunk of output is unique; if GWT is
074 * a target but those specific needs don't matter, use GWTRNG. ThrustAltRNG and
075 * MiniMover64RNG are both faster than DiverRNG usually (MiniMover64RNG is the fastest),
076 * but because they are unable to generate some outputs, that may make them a poor choice
077 * for some usage (ThrustAltRNG also has some bias toward specific numbers and produces
078 * them more frequently, but not frequently enough to make it fail statistical tests, and
079 * ThrustAltRNG can skip around in its output sequence like LightRNG). {@link JabRNG} is
080 * a mediocre choice for quality but is tied for the fastest with MiniMover64RNG.
081 * <br>
082 * There are many more RandomnessSource implementations! You might want significantly less
083 * predictable random results, which  {@link IsaacRNG} can provide, along with a
084 * large period. The quality of {@link PermutedRNG} is also good, usually, and it
085 * has a sound basis in PCG-Random, an involved library with many variants on its
086 * RNGs.
087 * <br>
088 * There may be reasons why you would want a random number generator that uses 32-bit
089 * math instead of the more common 64-bit math, but using a 32-bit int on desktop and
090 * Android won't act the same as that same 32-bit int on GWT. Since GWT is stuck with
091 * JavaScript's implementation of ints with doubles, overflow (which is needed for an
092 * RNG) doesn't work with ints as expected, but does with GWT's implementation of longs.
093 * If targeting GWT, {@link Lathe32RNG} is significantly faster at producing int values
094 * than any long-based generator, and will produce the same results on GWT as on desktop
095 * or Android (not all 32-bit generators do this). {@link Starfish32RNG} goes one step
096 * further than Lathe32RNG at an even distribution, and has better quality, but is
097 * slightly slower. While Lathe32RNG can produce all ints over the course of its period,
098 * it will produce some pairs of ints, or longs, more often than others and will never
099 * produce some longs. Starfish32RNG will produce all longs but one. {@link Oriole32RNG}
100 * and {@link XoshiroStarPhi32RNG} are also GWT-safe, but other generators that were
101 * thought to be GWT-friendly are not. These GWT-unsafe generators have other uses, but
102 * should not be used on GWT: {@link PintRNG} and {@link FlapRNG}. All other generators
103 * use longs, and so will be slower than the recommended Starfish32RNG or Lathe32RNG on GWT,
104 * but much faster on 64-bit JREs.
105 * @author Eben Howard - http://squidpony.com - howard@squidpony.com
106 * @author Tommy Ettinger
107 * @author smelC
108 */

190        /**
191         * Generates the next pseudorandom number. Subclasses should
192         * override this, as this is used by all other methods.
193         * <p>
194         * <p>The general contract of {@code next} is that it returns an
195         * {@code int} value and if the argument {@code bits} is between
196         * {@code 1} and {@code 32} (inclusive), then that many low-order
197         * bits of the returned value will be (approximately) independently
198         * chosen bit values, each of which is (approximately) equally
199         * likely to be {@code 0} or {@code 1}. The method {@code next} is
200         * implemented by class {@code Random} by atomically updating the seed to
201         * <pre>{@code (seed * 0x5DEECE66DL + 0xBL) & ((1L << 48) - 1)}</pre>
202         * and returning
203         * <pre>{@code (int)(seed >>> (48 - bits))}.</pre>
204         *
205         * This is a linear congruential pseudorandom number generator, as
206         * defined by D. H. Lehmer and described by Donald E. Knuth in
207         * <i>The Art of Computer Programming,</i> Volume 3:
208         * <i>Seminumerical Algorithms</i>, section 3.2.1.
209         *
210         * @param bits random bits
211         * @return the next pseudorandom value from this random number
212         * generator's sequence
213         * @since 1.1
214         */
215        @Override
216        protected int next(int bits) {
217            return randomnessSource.next(bits);
218        }

227         * value from this random number generator's sequence

293     * provided range, with min being inclusive and max being exclusive. It will
294     * sample the number of times passed in as the third parameter.
295     * <p>

351     * <p>
352     * Requires iterating through a random amount of the elements in set, so performance depends on the size of set but

426    /**
427     * Given a {@link List} l, this selects a random element of l to be the first value in the returned list l2. It
428     * retains the order of elements in l after that random element and makes them follow the first element in l2, and
429     * loops around to use elements from the start of l after it has placed the last element of l into l2.
430     * <br>
431     * Essentially, it does what it says on the tin. It randomly rotates the List l.
432     * <br>
433     * If you only need to iterate through a collection starting at a random point, the method getRandomStartIterable()
434     * should have better performance. This was GWT incompatible before GWT 2.8.0 became the version SquidLib uses; now
435     * this method works fine with GWT.
436     *
437     * @param l   A {@link List} that will not be modified by this method. All elements of this parameter will be
438     *            shared with the returned List.
439     * @param <T> No restrictions on type. Changes to elements of the returned List will be reflected in the parameter.
440     * @return A shallow copy of {@code l} that has been rotated so its first element has been randomly chosen
441     * from all possible elements but order is retained. Will "loop around" to contain element 0 of l after the last
442     * element of l, then element 1, etc.
443     */
444    public <T> List<T> randomRotation(final List<T> l) {
445        final int sz = l.size();
446        if (sz == 0)
447            return new ArrayList<>(0);
448
449                /*
450                 * Collections.rotate should prefer the best-performing way to rotate l,
451                 * which would be an in-place modification for ArrayLists and an append
452                 * to a sublist for Lists that don't support efficient random access.
453                 */
454        List<T> l2 = new ArrayList<>(l);
455        Collections.rotate(l2, nextInt(sz));
456        return l2;
457    }

459    /**
460     * Get an Iterable that starts at a random location in list and continues on through list in its current order.
461     * Loops around to the beginning after it gets to the end, stops when it returns to the starting location.
462     * <br>
463     * You should not modify {@code list} while you use the returned reference. And there'll be no
464     * ConcurrentModificationException to detect such erroneous uses.
465     *
466     * @param list A list <b>with a constant-time {@link List#get(int)} method</b> (otherwise performance degrades).
467     * @return An {@link Iterable} that iterates over {@code list} but start at
468     * a random index. If the chosen index is {@code i}, the iterator
469     * will return:
470     * {@code list[i]; list[i+1]; ...; list[list.length() - 1]; list[0]; list[i-1]}
471     */
472    public <T> Iterable<T> getRandomStartIterable(final List<T> list) {
473        final int sz = list.size();
474        if (sz == 0)
475            return new ArrayList<>(0);
476
477                /*
478                 * Here's a tricky bit: Defining 'start' here means that every Iterator
479                 * returned by the returned Iterable will have the same iteration order.
480                 * In other words, if you use more than once the returned Iterable,
481                 * you'll will see elements in the same order every time, which is
482                 * desirable.
483                 */
484        final int start = nextInt(sz);
485
486        return new Iterable<T>() {
487            @Override
488            public Iterator<T> iterator() {
489                return new Iterator<T>() {
490
491                    int next = -1;
492
493                    @Override
494                    public boolean hasNext() {
495                        return next != start;
496                    }
497
498                    @Override
499                    public T next() {
500                        if (next == start)
501                            throw new NoSuchElementException("Iteration terminated; check hasNext() before next()");
502                        if (next == -1)
503                                        /* First call */
504                            next = start;
505                        final T result = list.get(next);
506                        if (next == sz - 1)
507                                        /*
508                                         * Reached the list's end, let's continue from the list's
509                                         * left.
510                                         */
511                            next = 0;
512                        else
513                            next++;
514                        return result;
515                    }
516
517                    @Override
518                    public void remove() {
519                        throw new UnsupportedOperationException("Remove is not supported from a randomStartIterable");
520                    }
521
522                    @Override
523                    public String toString() {
524                        return "RandomStartIterator at index " + next;
525                    }
526                };
527            }
528        };
529    }

545     * Shuffle an array using the Fisher-Yates algorithm and returns a shuffled copy.
546     * GWT-compatible since GWT 2.8.0, which is the default if you use libGDX 1.9.5 or higher.
547     * <br>
548     * <a href="https://en.wikipedia.org/wiki/Fisher%E2%80%93Yates_shuffle">Wikipedia has more on this algorithm</a>.
549     *
550     * @param elements an array of T; will not be modified
551     * @param <T>      can be any non-primitive type.
552     * @return a shuffled copy of elements
553     */
554    @Override
555    public <T> T[] shuffle(final T[] elements) {
556        final int size = elements.length;
557        final T[] array = Arrays.copyOf(elements, size);
558        for (int i = size; i > 1; i--) {
559            swap(array, i - 1, nextIntHasty(i));
560        }
561        return array;
562    }

623     * <a href="https://en.wikipedia.org/wiki/Fisher%E2%80%93Yates_shuffle">Wikipedia has more on this algorithm</a>.

625     * @param <T>      can be any non-primitive type.
626     * @param buf a buffer as an ArrayList that will be filled with the shuffled contents of elements;
627     *            if null or non-empty, a new ArrayList will be allocated and returned
628     * @return a shuffled ArrayList containing the whole of elements in pseudo-random order.
629     */
630    @Override
631    public <T> ArrayList<T> shuffle(Collection<T> elements, /*@Nullable*/ ArrayList<T> buf) {
632        final ArrayList<T> al;
633        if (buf == null || !buf.isEmpty())
634            al = new ArrayList<>(elements);
635        else {
636            al = buf;
637            al.addAll(elements);
638        }
639        final int n = al.size();
640        for (int i = n; i > 1; i--) {
641            Collections.swap(al, nextInt(i), i - 1);
642        }
643        return al;
644    }

842     * Exclusive on bound (which must be positive), with an inner bound of 0.
843     * If bound is negative or 0 this always returns 0.
844     * <br>
845     * Credit for this method goes to <a href="https://oroboro.com/large-random-in-range/">Rafael Baptista's blog</a>,
846     * with some adaptation for signed long values and a 64-bit generator. This method is drastically faster than the
847     * previous implementation when the bound varies often (roughly 4x faster, possibly more). It also always gets
848     * exactly one random number, so it advances the state as much as {@link #nextInt(int)}.
849     * @param bound the outer exclusive bound; should be positive, otherwise this always returns 0L
850     * @return a random long between 0 (inclusive) and bound (exclusive)
851     */
852    public long nextLong(long bound) {
853        long rand = random.nextLong();
854        if (bound <= 0) return 0;
855        final long randLow = rand & 0xFFFFFFFFL;
856        final long boundLow = bound & 0xFFFFFFFFL;
857        rand >>>= 32;
858        bound >>= 32;
859        final long z = (randLow * boundLow >> 32);
860        final long t = rand * boundLow + z;
861        return rand * bound + (t >> 32) + ((t & 0xFFFFFFFFL) + randLow * bound >> 32) - (z >> 63);
862    }

1238    /**
1239     * Gets a random portion of data (an array), assigns that portion to output (an array) so that it fills as much as
1240     * it can, and then returns output. Will only use a given position in the given data at most once; uses the
1241     * Swap-Or-Not Shuffle to accomplish this without allocations. Internally, makes 1 call to {@link #nextInt()} and
1242     * 6 calls to {@link #nextSignedInt(int)}, regardless of the data being randomized. It will get progressively less
1243     * random as output gets larger.
1244     * <br>
1245     * Uses approximately the same code as {@link SwapOrNotShuffler}, but without any object or array allocations.
1246     *
1247     * @param data   an array of T; will not be modified.
1248     * @param output an array of T that will be overwritten; should always be instantiated with the portion length
1249     * @param <T>    can be any non-primitive type.
1250     * @return output, after {@code Math.min(output.length, data.length)} unique items have been put into it from data
1251     */
1252    @Override
1253    public <T> T[] randomPortion(T[] data, T[] output) {
1254        final int length = data.length, n = Math.min(length, output.length),
1255                func = nextInt(),
1256                a = nextSignedInt(length), b = nextSignedInt(length), c = nextSignedInt(length),
1257                d = nextSignedInt(length), e = nextSignedInt(length), f = nextSignedInt(length);
1258        int key, index;
1259        for (int i = 0; i < n; i++) {
1260            index = i;
1261            key = a - index;
1262            key += (key >> 31 & length);
1263            if(((func) + Math.max(index, key) & 1) == 0) index = key;
1264            key = b - index;
1265            key += (key >> 31 & length);
1266            if(((func >>> 1) + Math.max(index, key) & 1) == 0) index = key;
1267            key = c - index;
1268            key += (key >> 31 & length);
1269            if(((func >>> 2) + Math.max(index, key) & 1) == 0) index = key;
1270            key = d - index;
1271            key += (key >> 31 & length);
1272            if(((func >>> 3) + Math.max(index, key) & 1) == 0) index = key;
1273            key = e - index;
1274            key += (key >> 31 & length);
1275            if(((func >>> 4) + Math.max(index, key) & 1) == 0) index = key;
1276            key = f - index;
1277            key += (key >> 31 & length);
1278            if(((func >>> 5) + Math.max(index, key) & 1) == 0) index = key;
1279
1280            output[i] = data[index];
1281        }
1282        return output;
1283    }
